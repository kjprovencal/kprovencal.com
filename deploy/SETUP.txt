DigitalOcean droplet — one-time setup (run as root or sudo)
================================================================

1) Packages
   apt update && apt install -y nginx ufw certbot python3-certbot-nginx

2) Deploy directory (match GitHub secret DEPLOY_PATH, e.g. /var/www/kprovencal)
   mkdir -p /var/www/kprovencal/dist
   # Put the GitHub SSH user in the www-data group so it can share dist/ with nginx without world-readable files:
   #   usermod -aG www-data github-deploy
   # (log out/in or `newgrp www-data` so the extra group applies to SSH sessions.)
   # Own the tree by that user; use group www-data on dist/ so nginx can read via group permission:
   #   chown -R github-deploy:www-data /var/www/kprovencal
   # Optional setgid on dist (once): chmod g+s /var/www/kprovencal/dist — new subdirs may inherit group www-data.
   #
   # CI installs the API binary with `install -m 755 -o www-data -g www-data`, then `chgrp -R www-data dist/`
   # each deploy so nginx can read via the group bit (Vite output is usually 644/755 already).
   # Optional once on the server — setgid on dist so new subdirs inherit group www-data:
   #   sudo find /var/www/kprovencal/dist -type d -exec chmod g+s {} +
   #
   # Passwordless sudo for the deploy user (paths may vary): install, chgrp, rm, systemctl.
   # Easiest alternative: SSH_USER=root with the key in root’s authorized_keys.

3) API environment (secrets — never commit real values)
   nano /etc/personal-site-api.env
   Required at minimum:
     SESSION_SECRET=<32+ random bytes>
     ADMIN_PASSWORD_BCRYPT=<output of: go run ./cmd/hashpassword 'your-password'>
   Recommended for production behind nginx + HTTPS:
     LISTEN_ADDR=127.0.0.1:8080
     APP_ENV=production
     TRUST_PROXY=1
     BADGER_PATH=/var/www/kprovencal/data/badger
   Optional:
     CORS_ORIGIN=https://your.domain.example   # only if HTML is served from another origin
   For same-origin (nginx proxies /api and /admin API paths), you usually omit CORS_ORIGIN.

4) systemd unit
   cp deploy/personal-site-api.service /etc/systemd/system/personal-site-api.service
   # Edit paths/User in the unit file if your DEPLOY_PATH or user differs.
   systemctl daemon-reload
   systemctl enable --now personal-site-api

5) nginx
   Copy deploy/nginx-site.conf.example, set server_name and paths, then:
   nginx -t && systemctl reload nginx

6) TLS (Let’s Encrypt)
   certbot --nginx -d your.domain.example

7) Firewall
   ufw allow OpenSSH
   ufw allow 'Nginx Full'
   ufw enable

8) SSH key for GitHub Actions (deploy user or root)
   On the droplet: ssh-keygen -t ed25519 -C "github-deploy" -f /root/.ssh/github_deploy -N ""
   Authorize for login: cat /root/.ssh/github_deploy.pub >> /root/.ssh/authorized_keys
   Paste the *private* key into GitHub repo secret SSH_PRIVATE_KEY.
   Secrets to set: SSH_HOST, SSH_USER, DEPLOY_PATH (e.g. /var/www/kprovencal), SSH_PRIVATE_KEY.

   If rsync fails with "Permission denied" on dist/: ensure DEPLOY_PATH (and dist/) is owned by
   the GitHub SSH user, not www-data.

9) ARM droplets
   If the droplet is aarch64, change GOARCH in .github/workflows/ci.yml (api Build release binary) from amd64 to arm64.

10) First deploy
   Push to main after secrets are set; CI uploads dist + API binary and runs rsync + systemctl restart.
